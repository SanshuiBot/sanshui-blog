# 现代密码学与零知识证明实践：从椭圆曲线到 zk-SNARK

密码学是数字世界的基石。从 HTTPS 到区块链，从 Signal 端到端加密到 ZK-Rollup，背后都是同一套数学原理。

这篇文章会带你从最基础的对称加密讲起，一路到椭圆曲线、数字签名，最后落地到当下最火的零知识证明（Zero-Knowledge Proof）——zk-SNARK 的工程实现。

## 一、密码学的三大分支

```
┌──────────────────────────────────────────┐
│  对称密码（Symmetric）                    │
│  └─ AES / ChaCha20 / Camellia            │
├──────────────────────────────────────────┤
│  非对称密码（Asymmetric）                 │
│  └─ RSA / ECC / 后量子（Kyber / Dilithium）│
├──────────────────────────────────────────┤
│  哈希函数（Hash）                         │
│  └─ SHA-2 / SHA-3 / BLAKE3 / Keccak       │
├──────────────────────────────────────────┤
│  协议层：TLS / Signal / SSH / ZK          │
└──────────────────────────────────────────┘
```

## 二、对称加密：AES 与 ChaCha20

### 2.1 AES 的核心：代换-置换网络

AES 是 NIST 2001 年标准化的对称分组密码。它对 128-bit 数据块进行 10/12/14 轮处理（分别对应 128/192/256 位密钥）。

每一轮包含 4 个操作：

1. **SubBytes**：S-Box 非线性代换
2. **ShiftRows**：行移位
3. **MixColumns**：列混合（GF(2^8) 矩阵乘法）
4. **AddRoundKey**：异或轮密钥

```python
# Python cryptography 库示例
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
from cryptography.hazmat.backends import default_backend
import os

key = os.urandom(32)  # AES-256
iv = os.urandom(16)   # GCM 模式下是 nonce

cipher = Cipher(algorithms.AES(key), modes.GCM(iv), backend=default_backend())
encryptor = cipher.encryptor()
ciphertext = encryptor.update(b"Hello, AES-GCM!") + encryptor.finalize()
tag = encryptor.tag  # 认证标签
```

### 2.2 GCM vs CBC vs ECB

| 模式 | 性质 | 是否安全 |
|------|------|---------|
| ECB | 块独立加密 | ❌ 不要用 |
| CBC | 异或前一密文 | ⚠️ 无认证 |
| CTR | 流密码模式 | ⚠️ 无认证 |
| GCM | AEAD | ✅ 推荐 |
| ChaCha20-Poly1305 | AEAD | ✅ 移动端首选 |

**AEAD（Authenticated Encryption with Associated Data）** 同时提供机密性和完整性，是现代加密的唯一选择。

### 2.3 ChaCha20：纯软件实现的高速流密码

ChaCha20 用 ARX（Add-Rotate-Xor）结构，在没有 AES-NI 指令集的设备上比 AES 快 3 倍。

```python
from cryptography.hazmat.primitives.ciphers.aead import ChaCha20Poly1305

aead = ChaCha20Poly1305(os.urandom(32))
nonce = os.urandom(12)
ciphertext = aead.encrypt(nonce, b"secret", associated_data=b"meta")
plaintext = aead.decrypt(nonce, ciphertext, b"meta")  # 验证 tag 后才返回
```

**associated_data** 是不加密但参与认证的元数据（如 HTTP header、版本号）。

## 三、哈希函数：单向摘要

### 3.1 三大性质

- **抗原像性**（Preimage Resistance）：给 h(x)，难求 x
- **第二抗原像性**（Second Preimage）：给 x，难求 x' ≠ x 使 h(x) = h(x')
- **抗碰撞性**（Collision Resistance）：难找任意 x, x' 使 h(x) = h(x')

### 3.2 BLAKE3：当代最快的密码学哈希

```python
import blake3

h = blake3.blake3(b"hello world").hexdigest()
# 9c6f...64 字符
```

BLAKE3 的特点：

- Merkle-tree 内建，支持并行
- 任意长度输出（用作 KDF / MAC / XOF）
- 在 AVX2 / AVX-512 CPU 上比 SHA-256 快 10 倍

### 3.3 SHA-3 / Keccak：海绵结构

SHA-3 与 SHA-2 算法家族无关，基于 Keccak 的海绵结构（Sponge Construction）。

```python
from hashlib import sha3_256

h = sha3_256(b"hello").hexdigest()
```

**优势**：

- 长度扩展攻击免疫
- 同一原语可派生 SHAKE、KMAC、cSHAKE

## 四、密钥派生：PBKDF2、Argon2、HKDF

把低熵密码扩展为高熵密钥需要 KDF。

### 4.1 密码哈希：Argon2

```python
from argon2 import PasswordHasher

ph = PasswordHasher()
hash = ph.hash("correct horse battery staple")
ph.verify(hash, "correct horse battery staple")  # True
```

Argon2id 是 [PHC](https://www.password-hashing.net/) 竞赛优胜者，能抵抗：

- 时间-内存权衡（TMTO）攻击
- GPU / ASIC 并行攻击
- 侧信道攻击

### 4.2 HKDF：从已有密钥派生多个子密钥

```python
from cryptography.hazmat.primitives.kdf.hkdf import HKDF
from cryptography.hazmat.primitives import hashes

hkdf = HKDF(
    algorithm=hashes.SHA256(),
    length=32,
    salt=None,
    info=b'tls13 handshake',
)
derived = hkdf.derive(b"input keying material")
```

TLS 1.3 用 HKDF 做密钥调度，把一个 master secret 派生出 client_write_key、server_write_key、IV 等。

## 五、椭圆曲线密码学（ECC）

### 5.1 为什么是椭圆曲线？

RSA 要达到 128-bit 安全强度，需要 3072-bit 密钥。而 ECC 用 256-bit 密钥就能达到同样强度。

| 对称强度 | RSA | ECC |
|---------|-----|-----|
| 80 bit  | 1024 | 160 |
| 128 bit | 3072 | 256 |
| 256 bit | 15360 | 512 |

### 5.2 曲线方程与群运算

椭圆曲线一般形式：

```
y² = x³ + ax + b  (over a prime field F_p)
```

加法运算的几何意义：曲线上两点 P、Q，过 P 和 Q 画直线，与曲线的第三交点关于 x 轴对称即为 P+Q。

```python
# 简化版点加
def point_add(P, Q, p, a):
    if P is None: return Q
    if Q is None: return P
    x1, y1 = P
    x2, y2 = Q
    if x1 == x2 and y1 != y2:
        return None  # 无穷远点
    if x1 == x2:
        m = (3 * x1**2 + a) * pow(2 * y1, -1, p) % p
    else:
        m = (y2 - y1) * pow(x2 - x1, -1, p) % p
    x3 = (m**2 - x1 - x2) % p
    y3 = (m * (x1 - x3) - y1) % p
    return (x3, y3)

# 标量乘法：k * G
def scalar_mult(k, G, p, a):
    result = None
    addend = G
    while k:
        if k & 1:
            result = point_add(result, addend, p, a)
        addend = point_add(addend, addend, p, a)
        k >>= 1
    return result
```

### 5.3 secp256k1：比特币用的曲线

```
y² = x³ + 7 (mod p)
p = 2²⁵⁶ − 2³² − 977
G = (0x79BE..., 0x483A...)
n = 0xFFFFFFFF... (群阶)
```

比特币、以太坊都用 secp256k1。**为什么中本聪选它而不是 NIST 推荐的 secp256r1？** 业界普遍怀疑 NIST 曲线的随机种子可能被 NSA 植入后门。

### 5.4 ECDH 密钥协商

```python
from cryptography.hazmat.primitives.asymmetric import ec

# Alice 生成私钥、公钥
alice_priv = ec.generate_private_key(ec.SECP384R1())
alice_pub = alice_priv.public_key()

# Bob 同理
bob_priv = ec.generate_private_key(ec.SECP384R1())
bob_pub = bob_priv.public_key()

# Alice 用自己的私钥 + Bob 的公钥计算出共享密钥
shared = alice_priv.exchange(ec.ECDH(), bob_pub)
```

即使中间人监听所有公钥交换，也无法计算出共享密钥——这就是**计算 Diffie-Hellman 问题**（CDH）的困难性保证。

## 六、数字签名：ECDSA 与 Ed25519

### 6.1 ECDSA 签名

```python
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.asymmetric import ec

private_key = ec.generate_private_key(ec.SECP256K1())
public_key = private_key.public_key()

signature = private_key.sign(b"message", ec.ECDSA(hashes.SHA256()))
public_key.verify(signature, b"message", ec.ECDSA(hashes.SHA256()))
```

**ECDSA 的"陷阱"**：

1. **k 重用攻击**：每次签名需要随机数 k，如果两次签名用同一 k，私钥可以从两次签名中算出来。这就是 PlayStation 3 私钥被破解的原因。
2. **可塑性攻击**：(r, s) 和 (r, -s mod n) 都能验证通过，曾经导致比特币交易被篡改。

### 6.2 Ed25519：现代签名的优选

```python
from cryptography.hazmat.primitives.asymmetric.ed25519 import Ed25519PrivateKey

private_key = Ed25519PrivateKey.generate()
public_key = private_key.public_key()

signature = private_key.sign(b"message")  # 不需要哈希算法
public_key.verify(signature, b"message")
```

**Ed25519 的优势**：

- 确定性签名（无随机数，免疫 k 重用攻击）
- 抵抗侧信道攻击（无分支）
- 速度快（1μs 签名 / 验证）
- 公钥只有 32 字节，签名只有 64 字节

Signal、SSH、Tor、WireGuard 都用 Ed25519。

### 6.3 Schnorr 签名与 Taproot

Schnorr 签名由于专利问题（2008 年到期），很长时间没被采用。比特币在 2021 年通过 Taproot 升级引入 Schnorr：

- **线性可聚合**：多签可以聚合成单签形式
- **更隐私**：多签地址和单签地址看起来一样
- **更小**：64 字节 vs ECDSA 70-72 字节

## 七、密钥交换协议：TLS 1.3 握手

TLS 1.3 把握手从 2-RTT 减到 1-RTT，0-RTT 模式下甚至可以 0-RTT。

```
Client                                          Server
  │ ──ClientHello (key_share=X)────────────────► │
  │                                               │
  │ ◄──ServerHello (key_share=Y)──────────────── │
  │ ◄──EncryptedExtensions────────────────────── │
  │ ◄──Certificate────────────────────────────── │
  │ ◄──CertVerify─────────────────────────────── │
  │ ◄──Finished───────────────────────────────── │
  │                                               │
  │ ──Finished─────────────────────────────────► │
  │                                               │
  │ ──Application Data (encrypted)─────────────► │
```

**密钥派生**：

```
Early Secret = HKDF-Extract(0, PSK)
Handshake Secret = HKDF-Extract(Derived(Early), ECDHE)
Master Secret = HKDF-Extract(Derived(Handshake), 0)
```

每一步都用 HKDF，密钥层层嵌套，前向安全（Forward Secrecy）自然成立。

## 八、零知识证明：从 Sudoku 到 zk-SNARK

### 8.1 什么是零知识证明？

证明者 P 向验证者 V 证明"我知道某个秘密 s"，但 V 学不到任何关于 s 的信息。

**三个性质**：

1. **完备性**（Completeness）：若 P 真知道 s，V 总会接受
2. **可靠性**（Soundness）：若 P 不知道 s，V 接受的概率可忽略
3. **零知识性**（Zero-Knowledge）：V 学不到 s 的任何信息

### 8.2 经典例子：Sudoku 解的 ZKP

P 声称知道一道数独的解。V 想验证，但 P 不想暴露解。流程：

1. P 把解的每个数字盖住（一个数字一张卡片）
2. V 随机选"行验证"或"列验证"或"九宫格验证"
3. P 翻开被选行的所有卡片
4. V 验证 1-9 是否齐全
5. 重复多次，每次重置布局

V 每次只能验证一行/列/格，但 P 不知道 V 会选哪个。如果 P 作弊，每次有 2/3 概率被抓住。重复 k 次，作弊不被发现概率 (1/3)^k。

### 8.3 zk-SNARK 的数学基础

zk-SNARK = Zero-Knowledge Succinct Non-interactive ARgument of Knowledge

**关键算法步骤**：

1. **计算 → 算术电路**：把要证明的计算转成 boolean / arithmetic circuit
2. **电路 → R1CS**：Rank-1 Constraint System，每个约束形如 `⟨a,s⟩ · ⟨b,s⟩ = ⟨c,s⟩`
3. **R1CS → QAP**：Quadratic Arithmetic Program，多项式化
4. **QAP → zk-SNARK**：用 trusted setup 生成的 SRS（Structured Reference String）做配对友好曲线上的椭圆曲线配对

### 8.4 工程实战：Circom + snarkjs

[Circom](https://docs.circom.io/) 是 DSL，snarkjs 是 JS 库。我们做一个"我知道 x 使 hash(x) = y"的 ZKP。

**1. 写电路 `hash.circom`**：

```circom
pragma circom 2.0.0;

include "node_modules/@circomlib/circuits/poseidon.circom";

template HashPreimage() {
    signal input preimage;
    signal output hash;

    component hasher = Poseidon(1);
    hasher.inputs[0] <== preimage;
    hash <== hasher.out;
}
```

**2. 编译**：

```bash
circom hash.circom --r1cs --wasm --sym
```

**3. Trusted Setup（Powers of Tau + Phase 2）**：

```bash
snarkjs powersoftau new bn128 12 pot_0000.ptau
snarkjs powersoftau contribute pot_0000.ptau pot_0001.ptau --name="First"
# ...
snarkjs groth16 setup hash.r1cs pot_final.ptau hash_0000.zkey
snarkjs zkey contribute hash_0000.zkey hash_final.zkey
```

**4. 生成证明**：

```javascript
import { wtns } from "snarkjs";

const input = { preimage: "12345" };
const { proof, publicSignals } = await snarkjs.groth16.fullProve(
    input, "hash_final.zkey", "hash_js/hash.wasm"
);
// publicSignals 是 hash 值
```

**5. 验证证明**：

```javascript
const vKey = await snarkjs.zKey.exportVerificationKey("hash_final.zkey");
const isValid = await snarkjs.groth16.verify(vKey, publicSignals, proof);
console.log(isValid);  // true
```

整个过程：

- **证明大小**：约 200 字节（Groth16）
- **验证时间**：< 10ms
- **隐私保证**：验证者只看到 hash 值，看不到 preimage

### 8.5 PLONK vs Groth16

| 性质 | Groth16 | PLONK |
|------|---------|-------|
| 证明大小 | 最小（~200B） | 中等（~400B） |
| 验证速度 | 快 | 较快 |
| Trusted Setup | 每个电路单独 setup | Universal Setup，可复用 |
| Prover 时间 | 快 | 较慢 |

实际项目中：

- **zkSync、Scroll** 用 PLONK 变体
- **Aztec** 用 PLONKish
- **Polygon zkEVM** 用 zkEVM 专用 proof system

## 九、后量子密码学：NIST 选择的算法

量子计算机能跑 Shor 算法，破解 RSA / ECC。NIST 在 2024 年正式标准化了三个后量子算法：

| 算法 | 用途 | 基于的困难问题 |
|------|------|--------------|
| ML-KEM (Kyber) | 密钥封装 | 模块格上的 LWE |
| ML-DSA (Dilithium) | 数字签名 | 同上 |
| SLH-DSA (SPHINCS+) | 数字签名 | 哈希函数 |

### 9.1 使用 liboqs 调用 Kyber

```c
#include <oqs/kem_kyber_768.h>

uint8_t pk[OQS_KEM_kyber_768_length_public_key];
uint8_t sk[OQS_KEM_kyber_768_length_secret_key];

OQS_KEM_kyber_768_keypair(pk, sk);

uint8_t ct[OQS_KEM_kyber_768_length_ciphertext];
uint8_t ss_enc[OQS_KEM_kyber_768_length_shared_secret];
OQS_KEM_kyber_768_encaps(ct, ss_enc, pk);

uint8_t ss_dec[OQS_KEM_kyber_768_length_shared_secret];
OQS_KEM_kyber_768_decaps(ss_dec, ct, sk);
// ss_enc == ss_dec 即共享密钥
```

### 9.2 混合加密：安全过渡

不能一夜之间切到后量子，业界推荐**混合 KEM**：把一个传统 ECDH 共享密钥和一个 Kyber 共享密钥做异或，得到最终密钥。这样即使其中一个被攻破，另一个仍然保护。

TLS 1.3 的 `X25519Kyber768Draft00` 就是这种混合方案，[Cloudflare](https://blog.cloudflare.com/pq-tls-rolling-deployment/) 已经在生产部署。

## 十、实战：构建一个端到端加密聊天

### 10.1 协议设计（Signal Protocol 简化版）

```
1. 双方长期身份密钥 (Identity Key)：Ed25519
2. 双方一次性 PreKey：Curve25519
3. ECDH 三次（Alice 的 IK 与 Bob 的 SPK、Alice 的 EK 与 Bob 的 SPK、Alice 的 EK 与 Bob 的 IK）
4. HKDF 派生 Root Key
5. 每条消息用 Chain Key 派生 Message Key
```

### 10.2 双棘轮（Double Ratchet）算法

```
Root Key ──► Chain Key 1 ──► Chain Key 2 ──► ...
                │              │
                ▼              ▼
            Msg Key 1     Msg Key 2
```

每次收到新 ECDH 公钥时触发 **DH 棘轮**，每次发送消息触发 **Symmetric 棘轮**。

这样保证：

- **前向安全**：私钥泄露不能解密历史消息
- **后向安全**：未来消息在 DH 棘轮后也被新密钥保护

### 10.3 实现：使用 libsignal

```javascript
import { SignalProtocolAddress, SessionBuilder, SessionCipher } from '@privacyresearch/libsignal-protocol-typescript';

// Alice 准备发送给 Bob
const bobAddress = new SignalProtocolAddress('bob', 1);
const sessionBuilder = new SessionBuilder(aliceStore, bobAddress);
await sessionBuilder.processPreKey(bobPreKeyBundle);

const cipher = new SessionCipher(aliceStore, bobAddress);
const ciphertext = await cipher.encrypt('Hello Bob!');

// Bob 解密
const aliceAddress = new SignalProtocolAddress('alice', 1);
const bobCipher = new SessionCipher(bobStore, aliceAddress);
const plaintext = await bobCipher.decryptWhisperMessage(ciphertext.body);
```

这就是 Signal、WhatsApp、Telegram（私密聊天）背后协议的工程实现。

## 十二、密码学的工程法则

1. **不要自己造密码学原语**：用 libsodium、cryptography、Ring，它们经过审计
2. **AEAD or Die**：永远用 AES-GCM / ChaCha20-Poly1305，不要裸 AES-CBC
3. **Key = Random + Long**：32 字节随机数，密码必须经 Argon2 拉伸
4. **Nonce Never Reused**：GCM 模式下重用 nonce 会泄漏明文和认证密钥
5. **Constant-Time Compare**：所有 HMAC / Tag 比较必须用恒定时间，否则时序攻击
6. **Forward Secrecy by Default**：每次会话独立 ECDH，不留长期密钥痕迹
7. **Zeroize Secrets**：用完即擦，许多漏洞都是密钥在内存中残留被 dump

## 结语

密码学是少数几个"数学完美但工程脆弱"的领域。一段看似普通的代码，可能隐藏着 nonce 重用、时序攻击、密钥混淆这些致命漏洞。这就是为什么密码学库要经过 NIST、IETF、CRYPTO 会议的反复评审。

零知识证明是密码学过去 30 年最重要的进展之一。它让"证明而不暴露"成为可能，从匿名交易到隐私 AI 训练，从可验证计算到去中心化身份——ZK 正在重塑数字世界的信任模型。

掌握现代密码学，意味着你不仅能写出更安全的代码，还能理解整个数字信任体系是如何运转的。
