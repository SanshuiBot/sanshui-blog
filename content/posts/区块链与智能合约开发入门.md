# 区块链与智能合约开发入门：从 Solidity 到 DeFi

区块链从比特币白皮书到以太坊智能合约，再到如今 Rollup 与 ZK 大爆发，已经形成了一条完整的技术栈。这篇文章会从最底层的 EVM 开始讲起，带你完成一个真正能跑的 ERC-20 代币、NFT 合约，最后延伸到 DeFi 协议设计。

## 一、区块链到底是什么？

抛开"去中心化""不可篡改"这些营销词汇，区块链在技术上的本质是：**一个由全球数千个节点共同维护的、append-only 的状态机。**

```
创世状态 ──block 1──> 状态 S1 ──block 2──> 状态 S2 ──block 3──> ...
```

每一个区块包含若干交易，每笔交易都让状态机从一个状态跳到下一个状态。所有节点用共识算法（PoW / PoS / PBFT）保证看到的状态序列一致。

**公链 vs 联盟链**：

- 公链（Ethereum、Solana）：任何人都能加入网络、出块，靠经济激励保证安全
- 联盟链（Hyperledger Fabric、Quorum）：只有授权机构能出块，适合企业内部场景

本文聚焦公链开发，主要是 EVM 兼容链。

## 二、EVM：以太坊虚拟机

### 2.1 账户模型

以太坊有两类账户：

- **EOA（Externally Owned Account）**：由私钥控制，能发起交易
- **Contract Account**：由代码控制，被 EOA 调用时执行

每个账户都有：`nonce`、`balance`、`storageRoot`、`codeHash`。

### 2.2 Gas 与经济模型

每条 EVM 指令都有固定 Gas 消耗：

| 操作 | Gas | 说明 |
|------|-----|------|
| `ADD` | 3 | 加法 |
| `MUL` | 5 | 乘法 |
| `SSTORE`（新 slot） | 20000 | 写入新存储槽 |
| `SSTORE`（修改） | 5000 | 修改已有槽 |
| `SLOAD` | 800 | 读存储 |
| `CREATE` | 32000 | 创建合约 |

**为什么要 Gas？**

1. 防止 DoS：无限循环的合约会耗光 Gas 自动停止
2. 激励矿工/验证者：他们打包交易获得 Gas 费
3. 经济稀缺：每个区块的 Gas 上限（block gas limit）决定了能打包多少交易

**EIP-1559 之后的费用模型**：

```
total_fee = gas_used * (base_fee + priority_fee)
```

- `base_fee`：协议自动调整，被"烧掉"而不是给矿工
- `priority_fee`：用户给矿工的小费，拥堵时调高可以更快被打包

### 2.3 EVM 字节码与 Opcode

Solidity 编译后会生成 EVM 字节码。每个字节码对应一个 Opcode。比如：

```
PUSH1 0x80   // 把 0x80 压栈
PUSH1 0x40   // 把 0x40 压栈
MSTORE       // 在内存 0x40 处存 0x80（free memory pointer）
```

**调试技巧**：用 `evm.codes`（https://www.evm.codes/）查询 Opcode 的精确语义和 Gas 消耗。

## 三、Solidity 语法精要

### 3.1 一个最简单的合约

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

contract Counter {
    uint256 public count;

    event Counted(uint256 newCount, address indexed by);

    function increment() external {
        count += 1;
        emit Counted(count, msg.sender);
    }
}
```

**关键点**：

- `pragma` 指定编译器版本，`^0.8.24` 表示 0.8.24 到 0.9.0 之间
- `public` 自动生成 getter
- `external` 表示只能从外部调用（gas 更省）
- `event` + `emit` 用于记录日志，链下应用可监听

### 3.2 数据位置：storage、memory、calldata

```solidity
contract DataLocations {
    uint256[] public arr;        // storage：持久化
    mapping(address => uint256) public balances;  // storage

    function foo(uint256[] calldata input) external pure returns (uint256[] memory) {
        // calldata：只读、不可修改、gas 最省
        // memory：临时内存，函数结束销毁
        uint256[] memory output = new uint256[](input.length);
        for (uint256 i = 0; i < input.length; i++) {
            output[i] = input[i] * 2;
        }
        return output;
    }
}
```

### 3.3 可见性修饰符

| 修饰符 | 合约内部 | 子合约 | 外部调用 |
|--------|---------|--------|----------|
| `private` | ✓ | ✗ | ✗ |
| `internal` | ✓ | ✓ | ✗ |
| `public` | ✓ | ✓ | ✓ |
| `external` | ✗ | ✗ | ✓ |

**默认是 `internal`**。请显式声明以避免歧义。

### 3.4 修饰器（Modifier）

```solidity
contract Owned {
    address public owner;

    constructor() {
        owner = msg.sender;
    }

    modifier onlyOwner() {
        require(msg.sender == owner, "Not owner");
        _;
    }
}

contract Bank is Owned {
    function withdraw() external onlyOwner {
        payable(msg.sender).transfer(address(this).balance);
    }
}
```

`_` 是占位符，表示被修饰的函数体插入到这里。

## 四、ERC-20：从零写一个代币

### 4.1 ERC-20 标准接口

```solidity
interface IERC20 {
    function totalSupply() external view returns (uint256);
    function balanceOf(address account) external view returns (uint256);
    function transfer(address to, uint256 amount) external returns (bool);
    function allowance(address owner, address spender) external view returns (uint256);
    function approve(address spender, uint256 amount) external returns (bool);
    function transferFrom(address from, address to, uint256 amount) external returns (bool);

    event Transfer(address indexed from, address indexed to, uint256 value);
    event Approval(address indexed owner, address indexed spender, uint256 value);
}
```

### 4.2 完整实现

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

contract MyToken is IERC20 {
    string public name;
    string public symbol;
    uint8 public decimals = 18;

    uint256 private _totalSupply;
    mapping(address => uint256) private _balances;
    mapping(address => mapping(address => uint256)) private _allowances;

    constructor(string memory name_, string memory symbol_, uint256 initialSupply) {
        name = name_;
        symbol = symbol_;
        _totalSupply = initialSupply * 10 ** uint256(decimals);
        _balances[msg.sender] = _totalSupply;
        emit Transfer(address(0), msg.sender, _totalSupply);
    }

    function totalSupply() external view returns (uint256) {
        return _totalSupply;
    }

    function balanceOf(address account) external view returns (uint256) {
        return _balances[account];
    }

    function transfer(address to, uint256 amount) external returns (bool) {
        _transfer(msg.sender, to, amount);
        return true;
    }

    function approve(address spender, uint256 amount) external returns (bool) {
        _allowances[msg.sender][spender] = amount;
        emit Approval(msg.sender, spender, amount);
        return true;
    }

    function allowance(address owner, address spender) external view returns (uint256) {
        return _allowances[owner][spender];
    }

    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        uint256 currentAllowance = _allowances[from][msg.sender];
        require(currentAllowance >= amount, "Allowance exceeded");

        _allowances[from][msg.sender] = currentAllowance - amount;
        _transfer(from, to, amount);
        return true;
    }

    function _transfer(address from, address to, uint256 amount) internal {
        require(_balances[from] >= amount, "Insufficient balance");
        _balances[from] -= amount;
        _balances[to] += amount;
        emit Transfer(from, to, amount);
    }
}
```

**为什么 `decimals = 18`？**

以太坊的最小单位是 wei，1 ETH = 10^18 wei。所以 ERC-20 代币习惯上也用 18 位小数，方便与 DEX、借贷协议集成。但也有 USDC（6 位）、WBTC（8 位）这类例外。

### 4.3 安全要点

- **整型溢出**：Solidity 0.8+ 默认有溢出检查，不需要 SafeMath
- **重入攻击**：用 [checks-effects-interactions 模式](https://fravoll.github.io/solidity-patterns/checks_effects_interactions.html)
- **Approval 攻击**：approve 多次可能被 front-run，建议用 [increaseAllowance / decreaseAllowance](https://docs.openzeppelin.com/contracts/4.x/api/token/erc20#IERC20)

## 五、NFT：ERC-721 与 ERC-1155

### 5.1 ERC-721 单一 NFT

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC721/ERC721.sol";
import "@openzeppelin/contracts/utils/Counters.sol";

contract MyNFT is ERC721 {
    using Counters for Counters.Counter;
    Counters.Counter private _tokenIds;

    constructor() ERC721("MyNFT", "MNFT") {}

    function mint(address recipient) public returns (uint256) {
        _tokenIds.increment();
        uint256 newItemId = _tokenIds.current();
        _mint(recipient, newItemId);
        return newItemId;
    }
}
```

### 5.2 ERC-1155：多 token 标准

ERC-1155 一个合约同时支持 FT（同质化代币）和 NFT，比 ERC-721 节省大量 Gas，特别适合游戏道具：

```solidity
import "@openzeppelin/contracts/token/ERC1155/ERC1155.sol";

contract GameItems is ERC1155 {
    uint256 public constant GOLD = 0;
    uint256 public constant SWORD = 1;
    uint256 public constant SHIELD = 2;

    constructor() ERC1155("https://game.example/api/item/{id}.json") {
        _mint(msg.sender, GOLD, 10 ** 18, "");  // 1 亿金币
        _mint(msg.sender, SWORD, 1, "");          // 1 把剑（NFT）
        _mint(msg.sender, SHIELD, 1, "");
    }
}
```

### 5.3 元数据与 IPFS

NFT 的"内容"通过 `tokenURI` 指向 JSON 元数据：

```json
{
  "name": "Sword #1",
  "description": "A legendary sword",
  "image": "ipfs://QmXYZ.../sword.png",
  "attributes": [
    { "trait_type": "Rarity", "value": "Legendary" },
    { "trait_type": "Damage", "value": 100 }
  ]
}
```

**最佳实践**：

- 用 IPFS / Arweave 永久存储图片和元数据
- 用 [Pinata](https://pinata.cloud/) 或自建 pinning service
- 不要用中心化 HTTP 服务器——一旦下线 NFT 就空了

## 七、DeFi 协议设计：AMM 与借贷

### 7.1 AMM（自动做市商）

Uniswap V2 的核心公式：

```
x * y = k
```

池子里有代币 X 数量 x，代币 Y 数量 y，乘积 k 不变。买入 dx 数量的 X 时：

```
x' = x + dx_in
y' = y - dy_out
满足 (x + dx_in) * (y - dy_out) = x * y
```

简化版 AMM 合约：

```solidity
contract SimpleAMM {
    IERC20 public tokenA;
    IERC20 public tokenB;
    uint256 public reserveA;
    uint256 public reserveB;

    function addLiquidity(uint256 amountA, uint256 amountB) external {
        tokenA.transferFrom(msg.sender, address(this), amountA);
        tokenB.transferFrom(msg.sender, address(this), amountB);
        reserveA += amountA;
        reserveB += amountB;
    }

    function swapAtoB(uint256 amountAIn) external returns (uint256) {
        tokenA.transferFrom(msg.sender, address(this), amountAIn);
        // x * y = k
        uint256 newReserveA = reserveA + amountAIn;
        uint256 newReserveB = (reserveA * reserveB) / newReserveA;
        uint256 amountBOut = reserveB - newReserveB;

        tokenB.transfer(msg.sender, amountBOut);
        reserveA = newReserveA;
        reserveB = newReserveB;
        return amountBOut;
    }
}
```

**真实 Uniswap V2 还要处理**：

- 0.3% 手续费
- LP token 发行与销毁
- 价格预言机（TWAP）
- 滑点保护

### 7.2 借贷：Compound 模型

借贷协议的核心是**利率模型**——利用率越高，利率越高，激励存款、抑制借款。

```solidity
contract SimpleLending {
    IERC20 public asset;
    mapping(address => uint256) public deposits;
    mapping(address => uint256) public borrows;
    uint256 public totalDeposits;
    uint256 public totalBorrows;

    function deposit(uint256 amount) external {
        asset.transferFrom(msg.sender, address(this), amount);
        deposits[msg.sender] += amount;
        totalDeposits += amount;
    }

    function borrow(uint256 amount) external {
        require(address(this).balance >= amount, "Insufficient liquidity");
        asset.transfer(msg.sender, amount);
        borrows[msg.sender] += amount;
        totalBorrows += amount;
    }

    function utilizationRate() public view returns (uint256) {
        if (totalDeposits == 0) return 0;
        return (totalBorrows * 1e18) / totalDeposits;
    }
}
```

利用率 80% 以下时利率缓慢上升，超过 80% 时利率陡升——这就是所谓的"拐点模型"。

## 八、常见攻击与防御

### 8.1 重入攻击（Reentrancy）

经典案例 The DAO。攻击者在 `withdraw` 还没更新余额时，通过 fallback 函数再次调用 `withdraw`。

**防御**：

```solidity
// 1. Checks-Effects-Interactions 模式
function withdraw() external {
    uint256 bal = balances[msg.sender];
    require(bal > 0, "No balance");
    balances[msg.sender] = 0;  // Effect 先执行
    (bool ok, ) = msg.sender.call{value: bal}("");
    require(ok, "Transfer failed");
}

// 2. 使用 ReentrancyGuard
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";
contract Safe is ReentrancyGuard {
    function withdraw() external nonReentrant { ... }
}
```

### 8.2 Front-Running 与 MEV

交易在 mempool 中可见，攻击者可以复制高价值交易并提高 gas 抢跑。

**防御**：

- 使用 [Flashbots Protect](https://docs.flashbots.net/flashbots-protect/overview) 提交交易
- AMM 用 commit-reveal 方案隐藏交易意图
- 用 [TWAP](https://docs.uniswap.org/concepts/protocol/oracle) 而不是 spot price 做预言机

### 8.3 整数溢出

Solidity 0.8+ 默认有溢出检查。如果用 `unchecked` 块节省 Gas，必须确认溢出不会发生。

## 九、L2 与 Rollup：扩展性的未来

以太坊主网 TPS 约 15，远不够用。L2 解决方案：

| 方案 | 类型 | 安全性 | 例子 |
|------|------|--------|------|
| Optimistic Rollup | 欺诈证明 | 1 周挑战期 | Optimism、Arbitrum |
| ZK Rollup | 有效性证明 | 密码学保证 | zkSync、StarkNet |
| Validium | 数据可用性链下 | 弱 | Immutable X |
| Volition | 混合 | 可选 | zkSync Lite |

**开发者视角**：大多数 L2 提供 EVM 等效环境（EVM Equivalent），可以直接用 Solidity + Hardhat 部署，只需要改 RPC URL 和 chain ID。

## 结语

智能合约开发与传统 Web 开发最大的不同在于**不可撤销**——部署后的合约不能修改 bug，一旦被攻击资金就没了。这要求开发者：

1. 用最小权限原则设计合约
2. 关键操作加 timelock 和 multisig
3. 上主网前必须经过专业审计

区块链行业鱼龙混杂，但底层技术（密码学、分布式系统、经济机制设计）本身是严谨而美丽的。希望这篇文章能成为你 Web3 旅程的起点。
